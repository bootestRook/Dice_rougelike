## Context

项目已有 GM 物理投骰调试界面和骰子材质管线：`GmDiceDefinition` 维护 GM 可选材质 ID 与中文名称，`GmDiceCtrl` 负责把材质应用到圆角 D6 模型，`GmDiceViewport` 提供已有的 3D SubViewport、相机、灯光和舞台环境。当前缺口不是材质生成，而是缺少一个从主菜单进入的集中浏览与单颗细查工具。

该功能属于 GM / debug UI，不改变正式战斗规则。项目规则中“骰面材质不作为当前常规骰面槽位”仍然成立；这里检查的是视觉材质资源和 GM 原型材质。

## Goals / Non-Goals

**Goals:**
- 在主菜单右上角提供浮动 `GM` 入口，点击后展示 GM 功能列表。
- 从功能列表打开 `骰子材质检查` 工具。
- 在材质检查界面展示所有当前 GM 可识别骰子材质，包括资源材质与程序回退材质。
- 提供可滚动的“橱柜”浏览体验，每个材质以独立骰子预览和中文名称呈现。
- 点击材质预览后，在游戏内可拖拽弹窗中单独检查该骰子。
- 单骰检查弹窗支持旋转、缩放、自动旋转、重置视角、点数显示切换和光照调节。
- 保持新增 UI 文案中文，并为自动化测试暴露稳定节点名称。

**Non-Goals:**
- 不实现新的正式美术资源生成。
- 不引入外部素材、字体、图标包或第三方运行时。
- 不修改战斗流程、结算规则、奖励池、铸骰安装规则。
- 不把材质重新加入正式骰面槽位。
- 不替换现有 `GM物理投骰` 入口；现有入口可保留。
- 不创建系统原生独立窗口，检查弹窗使用游戏内 Control 弹层。

## Decisions

### 1. 主菜单内构建浮动 GM 入口

只在 `MainPrototypeView._build_view()` 生成主菜单时创建浮动 `GM` 按钮和菜单。进入地图、战斗、奖励、铸骰、商店、GM 工具时，现有 `_clear_screen()` 会清掉主菜单节点，因此入口自然只存在于主菜单。

备选方案是在全局根节点常驻 GM 入口。该方案更容易误触到运行中流程，也不符合“只在主菜单”的确认结果。

### 2. 新建独立 GM 材质检查屏幕

新增 `GmDiceMaterialInspectorScreen`，通过新场景 `scenes/debug/GmDiceMaterialInspectorScreen.tscn` 从主菜单打开。场景 root 只绑定脚本，复杂 UI 在脚本中构建，符合项目对复杂 UI 避免大量手写 `.tscn` 的要求。

备选方案是扩展 `GmPhysicsDiceTestScreen`。该界面已有投骰、目标点数、替换骰子、运动调参等职责，继续堆叠会让材质检查和物理投骰耦合。

### 3. 复用 GM 材质定义作为展示来源

材质列表以 `GmDiceDefinition.get_material_options()` 为准，确保 `standard / iron / glass / crystal / bronze / gold / repro_*` 都能展示。资源材质直接加载 `.tres`，程序材质通过复用或抽出 `GmDiceCtrl` 的材质构建逻辑来得到一致外观。

备选方案是只扫描 `assets/materials/dice/*.tres`。这会漏掉 `standard / iron / glass`，不满足“最好都展示”。

### 4. 预览与弹窗使用独立轻量 3D SubViewport

橱柜卡片和弹窗各使用轻量 `SubViewportContainer` 预览骰子。橱柜中的预览保持固定角度和低交互成本；弹窗中的预览处理拖拽旋转、滚轮缩放和光照滑杆。

备选方案是把所有材质都实例化到一个大型 3D 场景中，再用相机浏览。这个方案视觉感强，但 UI 滚动、点击命中和单独弹窗复用成本更高。

### 5. 光照调节限制在检查弹窗内部

弹窗提供预设和少量关键参数：主光强度、主光方向、环境光强度、补光强度。调节只影响当前弹窗预览，不回写材质资源、不影响 GM 物理投骰场景。

备选方案是写入材质或环境资源。该方案容易污染正式资源，不适合作为检查工具默认行为。

## Risks / Trade-offs

- 多个 SubViewport 同时显示可能增加主菜单 GPU 压力 → 橱柜预览限制尺寸，必要时只让可见卡片更新，弹窗关闭时释放独立预览。
- 程序回退材质与 `GmDiceCtrl` 私有方法重复实现可能产生不一致 → 优先抽出或新增共享 helper，让 GM 投骰和材质检查使用同一套材质解析逻辑。
- 中文文本在当前部分文件中曾出现编码显示问题 → 新增文件保持 UTF-8，新增测试检查关键中文节点文本不含英文内部 ID。
- 拖拽旋转和滚轮缩放容易与 ScrollContainer 冲突 → 橱柜卡片不处理拖拽，只在弹窗预览区域吞掉对应输入。
- Headless 测试无法可靠验证真实 3D 渲染效果 → Debug 测试重点检查资源、节点、材质映射、控件存在和自动化快照；视觉验收仍可通过编辑器人工查看。
