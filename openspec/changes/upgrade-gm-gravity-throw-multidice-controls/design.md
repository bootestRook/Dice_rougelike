## Context

当前 GM“骰子重力投掷”sandbox 由 `GMGravityThrowSandbox.gd` 动态创建 UI 和 3D 场景。它已有 `SubViewport`、相机、地面、边界、`RigidBody3D` 骰子和中文入口，但控制方式仍是 6 个目标面按钮，每次只创建一颗骰子；投掷过程也主要在冻结刚体后通过 Tween 模拟落地、弹起和定向转面。

战斗界面已经有 `Dice3DThrowLayer.gd`，能一次创建多颗 `RigidBody3D` 骰子，使用物理重力、地面/边界/骰子间碰撞，再在物理静止或超时后校准规则层目标面。GM 工具可以吸收这套思路，但仍必须保持独立调试工具身份，不调用 `RollService`、`BattleController` 或修改 `RunState`。

## Goals / Non-Goals

**Goals:**

- 让 GM sandbox 支持选择 1 到 6 颗骰子一起投掷。
- 将旧的 6 个目标面按钮替换为 6 个点数输入框，默认值均为 `1`，并通过单独“投掷骰子”按钮触发。
- 投掷阶段使用确定性的分段物理感演出，保留 `RigidBody3D` 骰子节点和碰撞体表现，并恢复旧版单骰的短促落地/弹起节奏。
- 目标面校准只在骰子落地并进入弹起/滚动阶段后接管。
- 当目标姿态差异较大时，让校准旋转伴随小幅水平滚动，避免原地转面。
- 提供摄像机视角按钮，可在侧面观察投掷弧线和上方观察落点分布之间切换。
- 用 Debug 测试覆盖 UI 控件、输入读取、多骰刚体、碰撞阶段标记和最终目标面。

**Non-Goals:**

- 不改变正式战斗投骰、重投、算分、奖励或铸骰规则。
- 不让物理结果决定正式骰面、骰型或分数。
- 不新增 D4 / D8 或更多骰子面数玩法。
- 不引入新的第三方依赖或正式美术资源。
- 不重写主页 GM 菜单结构。

## Decisions

### Decision: GM sandbox 自己实现轻量多骰物理，不直接复用战斗层节点

做法：在 `GMGravityThrowSandbox.gd` 内保留独立 `SubViewport` 和 GM 专用世界，将单个 `cube_body` 改为 `dice_bodies` 数组，按输入数量批量创建 `RigidBody3D`。物理参数、落点分布、静止阈值和校准流程可以参考 `Dice3DThrowLayer.gd`，但不直接实例化战斗 UI 层。

理由：GM 工具不需要战斗 view model、骰胚材质、目标槽位和收纳到 bench 的流程。独立实现能让测试入口更小，也能避免 GM 调试代码反向影响正式战斗。

替代方案：直接复用 `Dice3DThrowLayer.gd`。该方案可少写部分物理流程，但会要求 GM 构造战斗 view model 和 target rect，边界不够干净。

### Decision: 数量选择范围固定为 1 到 6，输入框固定 6 个

做法：UI 提供“投掷骰子数”选择控件，合法范围为 1 到 6；下方固定显示 6 个点数输入框，命名稳定。投掷时只读取前 N 个输入框，未参与的输入框保留值但不生成骰子。

理由：用户明确要求“6 个输入框，默认 6 个 1”。固定上限能保持 GM 工具简单，也匹配当前 D6 sandbox 的测试目标。

替代方案：输入框数量随骰子数量动态增删。该方案更灵活，但会增加 UI 和测试复杂度，且不符合“6 个输入框”的明确要求。

### Decision: 输入点数按 D6 目标面处理

做法：每个输入框接受 `1` 到 `6` 的整数，投掷前将非法值校正到合法范围；点数 `n` 对应目标 face index `n - 1`。这只是 GM 表现工具的目标面，不进入正式骰子数据模型。

理由：GM sandbox 的视觉骰面当前为标准 1 到 6 点，用户说的是“投掷点数”。按点数映射最直观，也便于 Debug 测试。

替代方案：允许输入 face index 或内部 ID。该方案更贴近代码，但玩家可见文本会暴露内部概念，不适合当前项目中文显示规范。

### Decision: 恢复旧版单骰分段节奏，落地弹起后才校准目标面

做法：投掷开始时创建 `RigidBody3D` 骰子、碰撞体、初速度和角速度记录，但视觉运动使用可控 Tween 分成两个阶段：约 `0.3s` 类重力曲线下落到地面并记录落地接触；随后约 `0.5s` 模拟触地后的弹起/滚动，并在弹起早段约 `0.2s` 内把朝向预旋到目标点数。目标面控制只在落地接触标记之后接管，空中下落阶段不校准朝向。

理由：需求强调“空中的碰撞，落地的碰撞”和“落地弹起的过程中，定向选择面的功能才接管”，同时旧版单骰的 `0.3s + 0.5s + 0.2s` 节奏明显更好看。确定性分段能保留物理感，又避免等待真实物理导致 GM 工具变慢、变散、难以调试。

替代方案：等待真实物理完整落地、反弹和趋稳后再校准。该方案更接近物理模拟，但多骰时容易拖慢节奏，且当前手感不如旧版单骰。

### Decision: 校准旋转带滚动偏移

做法：校准阶段计算当前旋转与目标旋转的差异；当差异超过阈值时，在旋转 Tween 中加入沿旋转方向的小幅水平位移和回落，形成“轻轻滚过去”的视觉。差异较小时只做微调，避免过度晃动。

理由：原地直接转到目标面会削弱物理可信度。小幅滚动能把“定向选择面”伪装成落地后的自然滚动。

替代方案：只旋转不移动。该方案更简单，但正是本次需求要避免的表现。

### Decision: 视角切换只调整 GM sandbox 相机

做法：在 GM sandbox header 增加“视角：侧面 / 视角：上方”按钮。侧面视角沿现有 Z 轴观察投掷弧线；上方视角把相机移到 Y 轴上方并向下看投掷台，用于观察多骰落点和碰撞分布。

理由：视角切换是调试观察能力，不应改变骰子物理、目标点数、校准或正式战斗表现。

替代方案：新增两个独立 GM sandbox。该方案入口更重，而且会重复投掷控制，不适合当前小工具。

## Risks / Trade-offs

- [Risk] 多骰确定性演出可能不如逐帧物理随机。→ Mitigation: 使用分散起点、假接触面、弹起高度和轻微滚动偏移制造物理反馈，同时保持节奏稳定。
- [Risk] 输入框非法值导致目标面不明确。→ Mitigation: 投掷前统一校正为 1 到 6，并在状态文本显示已使用的目标点数。
- [Risk] 投掷手感再次被改慢。→ Mitigation: Debug 测试固定验证落地约 `0.3s`、弹起约 `0.5s`、目标面预旋约 `0.2s` 和总演出时长。
- [Risk] 多骰在 GM 区域内互相遮挡。→ Mitigation: 使用分散起点和中心落点环，同时保持边界碰撞体不可见。
- [Risk] 复制战斗层逻辑造成维护重复。→ Mitigation: 只复用设计思路，必要时抽取非常小的纯几何辅助函数，避免引入战斗依赖。
