## Context

上一轮视觉变更已经能稳定生成整图截图、manifest 和基础报告，但截图对比证明它没有显著接近参考图。根因不是单个参数错误，而是工作流把“结构存在”当成“视觉完成”：只要节点、灯光、材质和测试存在，任务就能完成，即使肉眼效果仍然像半透明空壳骰子。

本轮变更以参考图为唯一视觉目标。参考图的核心视觉信号是：深蓝星盘舞台、蓝金 UI 框架、六颗实体圆角骰子、顶面大数字、前脸星徽、侧面厚度、彩色发光边、柔光高光、接触阴影和底部操作区。尤其骰子必须先从“透明外壳”变成“可落地的实体块”。

## Goals / Non-Goals

**Goals:**

- 建立参考图级视觉完成定义，避免 OpenSpec 任务只因工程结构存在而完成。
- 将视觉验收报告拆成工程状态和视觉接受状态。
- 优先修复骰子实体感：三面可见、实体圆角、顶面数字、前脸星徽、侧面暗部、边缘发光、接触阴影。
- 让灯光、Bloom、反射、SSAO 和台面材质服务于参考图效果，而不是作为独立微调。
- 每轮结算前展示改动前后截图，并记录人工判断。

**Non-Goals:**

- 不改变战斗规则、算分规则、奖励规则、骰面数据模型或玩家操作流程。
- 不把“骰面材质”重新作为正式常规骰面槽位。
- 不引入 C#、第三方运行时框架、版权不明外部素材或外部字体包。
- 不要求一次达到最终商业美术质量，但本轮每个完成项必须能在截图中看到明显改进。

## Decisions

### 1. 完成状态拆成工程完成和视觉接受

OpenSpec task 只能在工程检查通过后标为工程完成；视觉接受必须另行记录。最终任务完成前，报告必须包含改动前截图、改动后截图、参考图差距清单和人工判断。这样可以避免“测试全绿但画面不像”的结果被误判为完成。

替代方案是继续用 runner 的 valid manifest 作为完成标准。这个方案只能证明截图有效，不能证明视觉接近参考图。

### 2. 骰子实体感优先于灯光微调

下一轮实现应先处理骰子几何和表面层级，而不是继续调灯。当前半透明整壳式 rim 会盖住本体，导致空心感。新方案应把骰子拆成实体本体、倒角/边框高光、顶面数字贴花、前脸星徽贴花、状态层和接触阴影，且边缘高光不得用整颗透明外壳覆盖主体。

替代方案是继续叠加透明壳和 Bloom。它短期能变亮，但会让骰子更不像参考图中的实体块。

### 3. 用参考图差距清单驱动任务，而不是用文件清单驱动任务

每个任务必须对应一个可观察差距，例如“前脸星徽缺失”“侧面暗部不足”“骰子没有落在台面上”。文件和节点只是实现手段，不能作为唯一完成条件。

替代方案是继续把任务写成“添加节点”“新增灯光”。这种任务容易完成，但不保证截图变化。

### 4. 自动化验收负责可重复性，人工验收负责相似度

自动化仍然必须检查固定分辨率、固定相机、manifest、latest 清理、渲染特性、截图路径和基础内容阈值。相似度判断则由人工根据 Codex 对话中的截图对比决定。报告必须明确“视觉接受：通过 / 未通过 / 需下一轮”。

替代方案是尝试一次性用像素阈值判断像不像。由于参考图和 Godot 输出风格差异较大，纯像素阈值容易误判。可先加入基础统计，但不能替代人工验收。

### 5. 参考图构图作为整图 case 的主验收目标

整图 case 必须覆盖左侧比分面板、顶部遗物/道具区、中央星盘骰子区和底部行动栏。骰子区是第一优先级，但 UI 比例、蓝金边框和星盘透视也必须进入差距清单。

替代方案是只做局部骰子预览。局部预览适合调材质，但无法保证战斗画面整体接近参考图。

## Risks / Trade-offs

- 骰子层级增加可能提高节点数量和材质复杂度 → 先限制在视觉复刻/GM 场景，确认稳定后再接入正式战斗视口。
- Label3D/贴花/透明层可能产生深度排序问题 → 优先使用实体贴花网格或轻微 surface offset，并在多个角度截图里验证可读性。
- 自动化无法完全判断审美相似度 → 报告保留人工视觉接受状态，未接受时任务不得最终完成。
- 参考图很精致，单轮可能无法完全达到 → tasks 按“先骰子、再星盘/UI、再灯光后处理”的顺序设置，每轮必须有肉眼改进。

## Migration Plan

1. 保存当前失败截图作为新 baseline，并在报告中标注上一轮视觉未接受。
2. 建立新的参考图差距清单和视觉接受报告模板。
3. 先改骰子实体模型与面层，生成改动前后截图。
4. 再调星盘台面、UI 遮罩和构图比例，生成第二组截图。
5. 最后调灯光、反射、Bloom 和接触阴影，生成最终截图。
6. 若某轮截图倒退，回退该轮视觉参数或 builder 输出，保留验收流程和失败记录。

## Open Questions

- 顶面数字和前脸星徽最终采用 Label3D、贴花网格、独立 Mesh 还是预生成纹理，需要在实现时用截图比较可读性。
- 是否需要把参考图保存为本地受控资源，取决于用户是否提供可存档授权的原图文件。
- UI 边框是否复刻到完整交互界面，还是先只在视觉验收 overlay 中达到参考图比例，需要实现前确认接入范围。
