# 视觉复刻验收工作流

本文档备份当前项目的视觉复刻验收流程，用于 UI、美术、材质、灯光、渲染、场景构图相关任务。它补充 `AGENTS.md` 中的截图规则：视觉任务结算前必须在 Codex 对话中展示改动前后截图，辅助人工验收。

## 核心原则

- OpenSpec 任务完成不等于视觉达标。工程完成只表示资源、脚本、截图、测试和报告可运行；视觉接受必须由参考图对齐结果决定。
- 视觉复刻任务必须先锁定参考图、当前基线截图和目标差距，不能直接进入零散调参。
- 每轮只解决少数主因，优先处理肉眼最明显的差距。若骰子形体、面值、阴影或构图未达标，不应把轻微灯光变化算作主要完成。
- 自动化验收负责保证截图来源、分辨率、构图、manifest、资源有效和基础阈值；“像不像参考图”的结论必须保留人工判断。

## 标准流程

1. 保存基线：在改动前运行视觉验收截图，记录 run_id、截图路径和 manifest 状态。
2. 拆参考图：把目标画面拆成可观察项，例如骰子实体感、三面可见、顶面数字、前脸星徽、发光边框、接触阴影、星盘透视、蓝金 UI、Bloom 和反射层次。
3. 写硬门槛：OpenSpec tasks 不只写“添加节点/灯光/材质”，还要写肉眼结果，例如“骰子不得呈现空心壳感”“每颗骰子至少可见顶面、前面和侧面”。
4. 小步迭代：每轮只改一个主要视觉面，完成后立即截图对比。
5. 对话展示：任务结算前，在 Codex 对话中展示改动前、改动后或合成对比图。
6. 双状态收尾：报告中分别说明工程完成状态和视觉接受状态。若用户未接受视觉结果，OpenSpec 任务不得标为最终完成。

## 截图状态约束

- 截图文件名必须和画面状态一致。`actual`、`physical`、`settle`、`after_throw` 等命名只能用于已经真实触发目标流程并等待关键状态完成的截图。
- GM 投掷验收截图必须执行一次真实投掷，并在 `last_settled_face_value` 全部有效、`last_settled_position` 位于投掷面附近后再保存。只创建骰子、设置骰子数量或显示待机悬浮骰子的截图，不得命名为实际投掷截图。
- 截图 runner 必须在 manifest 中记录 `capture_phase`、输出路径、骰子数量、相机参数和关键状态字段。GM 投掷当前使用 `capture_phase = physical_settle_after_drop`。
- 如果为了对比保留待机图，文件名和报告必须明确写成 `idle`、`ready` 或 `staged`，不能混入“投掷后”验收结论。
- 对话展示前要核对图片路径、文件修改时间和 manifest。若实际运行画面与发出的截图不一致，优先检查截图脚本是否真的进入了目标状态，而不是继续调视觉参数。

## 人工验收清单

- 改动后截图是否明显比改动前更接近参考图。
- 骰子是否是实体圆角骰子，而不是半透明空壳。
- 骰子是否有清晰的顶面数字、前脸星徽、侧面暗部和发光边框。
- 骰子是否稳定落在星盘上，有接触阴影和反射/高光关系。
- 星盘是否保留深蓝底、金色环线、星点、径向刻线和舞台透视。
- UI 是否维持中文、蓝金边框、左右/顶部/底部布局和参考图比例。
- Bloom、Glow、反射是否增强质感，而不是洗白数字或吞掉边缘。

## 完成定义

- 工程完成：OpenSpec artifacts 完整，相关测试通过，视觉验收 runner 生成有效 manifest 和截图。
- 视觉接受：改动前后对比已展示，参考图差距清单逐项更新，用户或人工验收者确认结果达到本轮目标。
